# Magical Competition 要件定義書

## 概要

「マジカルコンペ（Magical Competition）」は、魔石カードを巧みに組み合わせて誰よりも早く全カードを使い切ることを目指す2Dカードゲームのデジタル版。1人用（対戦相手はAI）としてWebGLで動作する。初期バージョンでは魔石カードのみの基本ルールを実装し、魔法カード（魔導書）は後続バージョンで追加する。

## 機能要件（EARS記法）

**【信頼性レベル凡例】**:

### 通常要件（ゲームセットアップ）

- **REQ-001**: システムは、ゲーム開始時に45枚の魔石カード（5色×9番号）をシャッフルし、全プレイヤーに均等配布しなければならない 🔵 *マニュアル「遊ぶ準備①」より*

- **REQ-002**: システムは、プレイヤーがAI対戦相手の人数を1〜4体から選択できるようにしなければならない 🔵 *ユーザヒアリングより*

- **REQ-003**: システムは、各プレイヤーの配布されたカードを裏向きの山札として配置しなければならない 🔵 *マニュアル「遊ぶ準備②」より*

- **REQ-004**: システムは、ゲーム開始時に各プレイヤーが自分の山札から3枚を手札として引くようにしなければならない 🔵 *マニュアル「遊ぶ準備③」より*

- **REQ-005**: システムは、ゲーム開始時に場札を「好きな色の5」として疑似的に設定しなければならない（実際のカードは出さず、場札の状態のみ設定する） 🔵 *ユーザヒアリングより*

### 通常要件（ゲームプレイ）

- **REQ-101**: システムは、プレイヤーが場札と同じ数字のカードを1〜3枚同時に出せるようにしなければならない 🔵 *マニュアル「遊び方①」より*

- **REQ-102**: システムは、プレイヤーが場札と同じ色のカードを1〜3枚同時に出せるようにしなければならない 🔵 *マニュアル「遊び方①」より*

- **REQ-103**: システムは、プレイヤーが2〜3枚のカードを足し算・引き算して一の位を場札の数字と一致させて出せるようにしなければならない 🔵 *マニュアル「遊び方①」より*

- **REQ-104**: システムは、REQ-103の計算過程でマイナスの数字を禁止し、0は許容しなければならない 🔵 *マニュアル「遊び方①」+ユーザヒアリングより*

- **REQ-105**: システムは、カードを出した後の場札参照を、最後に出したカードの色と数字に更新しなければならない（同数字複数枚・同色複数枚・計算出しすべて共通） 🔵 *ユーザヒアリングより*

- **REQ-106**: システムは、カードを出せない場合にパスを宣言し、手札から任意の枚数（0枚も可）を山札の一番下に入れる操作を提供しなければならない 🔵 *マニュアル「遊び方①」+ユーザヒアリングより*

- **REQ-107**: システムは、全プレイヤーが連続でパスした場合、場札をリセットし、場札を「好きな色の5」として疑似的に設定して再開しなければならない（実際のカードは出さない） 🔵 *マニュアル「遊び方①」+ユーザヒアリングより*

- **REQ-108**: システムは、カードを出した後に手札が3枚になるまで山札からカードを補充しなければならない 🔵 *マニュアル「遊び方②」より*

- **REQ-109**: システムは、山札が尽きた場合は3枚に足りない状態でも補充を終了し、リーチを宣言しなければならない 🔵 *マニュアル「遊び方②」より*

- **REQ-110**: システムは、手札と山札がすべて無くなったプレイヤーをそのラウンドの勝者としなければならない 🔵 *マニュアル「遊び方③」より*

### 通常要件（勝利・結果）

- **REQ-201**: システムは、1ラウンド制で勝敗を決定しなければならない 🔵 *ユーザヒアリングより*

- **REQ-202**: システムは、ラウンド終了時に勝者の得点として他プレイヤーの手札＋山札の合計枚数を算出し、ゲーム内ダイアログで表示しなければならない 🔵 *マニュアル「遊び方④」+ユーザヒアリングより*

### 通常要件（AI）

- **REQ-301**: システムは、Normal難易度のAIを実装しなければならない。AIは基本的な戦略（多く出せる手を優先、不要カードの戦略的パス）に基づいて行動する 🔵 *ユーザヒアリングより（Normal のみ）*

- **REQ-302**: システムは、AIのターンを自動的に実行し、思考中であることをUIで示さなければならない 🔵 *ユーザヒアリングより*

### 通常要件（UI・画面）

- **REQ-401**: システムは、タイトルシーンとゲームシーンの2画面構成としなければならない 🔵 *ユーザヒアリングより*

- **REQ-402**: システムは、タイトルシーンでAI対戦人数の選択とゲーム開始ボタンを提供しなければならない 🔵 *REQ-002+REQ-401から導出*

- **REQ-403**: システムは、ゲームシーンで以下のUI要素をCanvas上に配置しなければならない：プレイヤー手札、場札エリア、山札（残り枚数表示）、各AIの手札枚数・山札枚数表示 🔵 *ユーザヒアリングより*

- **REQ-404**: システムは、ゲーム結果をゲーム内ダイアログで表示し、タイトルに戻るボタンを提供しなければならない 🔵 *ユーザヒアリングより*

### 条件付き要件

- **REQ-501**: カードが均等に配布できない場合、システムは余りカードをゲームから除外しなければならない 🔵 *マニュアル「遊ぶ準備①」より*

- **REQ-502**: プレイヤーの山札が空になった場合、システムは手札が3枚未満でも補充を行わず、リーチ状態として表示しなければならない 🔵 *マニュアル「遊び方②」より*

### 制約要件

- **REQ-601**: システムは、uGUI Canvas（Screen Space - Overlay）を使用してすべてのUI要素を描画しなければならない 🔵 *tech-stack.md より*

- **REQ-602**: システムは、MVC + State Machine + ScriptableObject アーキテクチャに従って実装しなければならない 🔵 *CLAUDE.md・tech-stack.md より*

- **REQ-603**: システムは、WebGL ビルドとして動作しなければならない 🔵 *tech-stack.md より*

## 非機能要件

### パフォーマンス

- **NFR-001**: システムは、WebGL環境で60fps安定で動作しなければならない 🔵 *tech-stack.md「品質基準」より*

- **NFR-002**: システムは、カードアニメーション（移動・反転）を30fps以下に落とさず実行しなければならない 🔵 *ユーザヒアリングより*

- **NFR-003**: システムは、AIの思考処理を1ターンあたり500ms以内に完了しなければならない 🔵 *ユーザヒアリングより*

- **NFR-004**: システムは、WebGLビルドサイズを可能な限り軽量化しなければならない（目標: 初期ロード30MB以下） 🔵 *ユーザヒアリングより*

### ユーザビリティ

- **NFR-101**: システムは、プレイヤーの手札をタップ/クリックで選択可能な状態で視覚的に表示しなければならない 🔵 *ユーザヒアリングより*

- **NFR-102**: システムは、出せないカードをグレーアウトして出せるカードと視覚的に区別しなければならない 🔵 *ユーザヒアリングより*

- **NFR-103**: システムは、現在のターンプレイヤーを明示的に表示しなければならない 🔵 *ユーザヒアリングより*

- **NFR-104**: システムは、場札の現在の数字と色を明確に表示しなければならない 🔵 *ユーザヒアリングより*

### 対応ブラウザ

- **NFR-201**: システムは、Chrome, Firefox, Edge の最新版で動作しなければならない 🔵 *tech-stack.md「品質基準」より*

### テスト

- **NFR-301**: Core/Systems のゲームロジックは80%以上のテストカバレッジを確保しなければならない 🔵 *tech-stack.md・CLAUDE.md より*

## エッジケース

### カード配布

- **EDGE-001**: 2人対戦時、45枚を2人に配布すると22枚ずつ+1枚余り。余り1枚はゲームから除外される 🔵 *マニュアル「遊ぶ準備①」より*

- **EDGE-002**: 5人対戦時、45枚を5人に配布すると9枚ずつ+0枚余り。余りなし 🔵 *マニュアルより算出*

- **EDGE-003**: 3人対戦時、45枚÷3=15枚ずつ。余りなし 🔵 *マニュアルより算出*

- **EDGE-004**: 4人対戦時、45枚÷4=11枚ずつ+1枚余り 🔵 *マニュアルより算出*

### ゲームプレイ

- **EDGE-101**: 手札が1枚のみで山札も空の場合、その1枚を出せれば勝利。出せなければパスして0枚を山札に入れる（手札1枚のまま） 🔵 *ユーザヒアリングより*

- **EDGE-102**: 手札が0枚で山札にカードが残っている場合、補充フェーズで3枚まで引く。手札0枚は通常到達しない（出した後に補充するため） 🔵 *ユーザヒアリングより*

- **EDGE-103**: 同じ数字3枚出しで手札が0枚になり、山札も空の場合、その時点で即座に勝利 🔵 *ユーザヒアリングより*

- **EDGE-104**: パス時に手札を全て山札に戻すことが可能（0枚も可なので、逆に全枚数も可） 🔵 *マニュアル「任意の枚数」の解釈*

- **EDGE-105**: 全員パスでリセット後、場札を「好きな色の5」として疑似設定する。実際のカードを出す必要がないため、手札に5がなくても問題なし 🔵 *ユーザヒアリングより解消*

### 計算ルール

- **EDGE-201**: 足し算のみの組み合わせ：例）場札7、手札 [3,4] → 3+4=7 ✓ 🔵 *マニュアルより*

- **EDGE-202**: 引き算を含む組み合わせ：例）場札7、手札 [9,2] → 9-2=7 ✓ 🔵 *マニュアルより*

- **EDGE-203**: 3枚の組み合わせ：例）場札3、手札 [5,1,7] → 5+1-7=-1 ✗（マイナス）、5-1+7=11→1の位=1 ✗、7-5+1=3 ✓ 🔵 *マニュアルより*

- **EDGE-204**: 計算結果が0の場合：例）場札が一の位0（=10）のケースは存在しない（カードは1〜9のため）。計算途中の0は許容 🔵 *ユーザヒアリングより*

- **EDGE-205**: 同じ計算結果に到達する複数の組み合わせが存在する場合、プレイヤーが使用するカードを選択する 🔵 *ユーザヒアリングより*

### 場札の更新

- **EDGE-301**: 同じ数字のカードを複数枚出した場合、場札の数字は変わらないが色は最後に出したカードの色に更新される 🔵 *ユーザヒアリングより（REQ-105: 最後の1枚が場札）*

- **EDGE-302**: 同じ色のカードを複数枚出した場合、場札の色は変わらないが数字は最後に出したカードの数字に更新される 🔵 *ユーザヒアリングより（REQ-105: 最後の1枚が場札）*

- **EDGE-303**: 計算で出した場合、場札は最後に出したカードの数字と色に更新される（計算結果ではなくカードそのものの値） 🔵 *ユーザヒアリングより*

### AI特有

- **EDGE-401**: AIが計算出しを判断する際、全ての有効な2〜3枚の組み合わせを全探索する 🔵 *ユーザヒアリングより*

- **EDGE-402**: 2人対戦でAIが魔法カード対象を選ぶ場合、自動的にプレイヤーが対象になる（初期版では魔法カード未実装のため該当なし） 🔵 *ユーザヒアリングより*
