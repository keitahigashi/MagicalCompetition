- カード出し後に手札が減少する
- 場札が更新される
- Draw フェーズで手札が補充される

### 3. パス〜全員パスリセットテスト 🔵

**信頼性**: 🔵 *architecture.md テスト戦略より*

### 4. 勝利テスト 🔵

## PlayModeテスト要件

### 1. ゲーム初期化: 全プレイヤーに手札3枚・場札設定・正しいフェーズ 🔵

ゲームを初期化した際、全プレイヤーに手札が3枚ずつ配布され、場札が設定され、正しいフェーズ（Play）に遷移することを確認する。

### 2. カード出しフロー: 手札減少・場札更新・手札補充の連続動作 🔵

カードを出した際、手札が減少し、場札が更新され、Drawフェーズで手札が補充される一連の動作を確認する。

### 3. パスフロー: パスカウント増加・次プレイヤーへ 🔵

パスを選択した際、パスカウントが増加し、次のプレイヤーのターンに遷移することを確認する。

### 4. 全員パスリセット: 場札リセット・色選択から再開 🔵

全プレイヤーがパスした際、場札がリセットされ、SelectFieldCardフェーズから再開することを確認する。

### 5. 勝利フロー: 全カード使い切り→結果表示 🔵

プレイヤーが手札と山札を全て使い切った際、Endフェーズに遷移し、ResultDialogViewが表示されることを確認する。

**テストファイル**: `Assets/_Project/Tests/PlayMode/Integration/GameFlowIntegrationTests.cs`

## 実装手順

### TDD手順

1. **tdd-requirements**: architecture.mdからゲームフロー仕様、各フェーズの遷移条件を確認する
2. **tdd-testcases**: 上記5つのテストケースをGameFlowIntegrationTests.csに記述する
3. **tdd-red**: テストを実行し、全てREDであることを確認する
4. **tdd-green**: テストが通るように統合環境を調整し、全テストをGREENにする
   - テスト用のGameScene構築（テスト専用のセットアップ処理）
   - 各フェーズ遷移のawait/yield対応
   - AIアクションのモック化またはタイムアウト設定
   - ResultDialogViewの表示確認
5. **tdd-refactor**: テストヘルパークラスの作成、テストデータのファクトリ化
6. **tdd-verify-complete**: 全テストがパスし、完了条件を満たすことを確認する
